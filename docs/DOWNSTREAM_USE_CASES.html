1. Tier 1 — Works Today With the Bundle Alone ship-ready

Thirteen use cases powered entirely by the 2.8 MB bundle — no external data, no retraining. Each card shows a full workflow.

1Site Selection — Brand ExpansionGCN-64●
Retail / F&Bsimilarityinteractive
Who
VP Expansion, head of real estate at a brand (cafe, QSR, fitness, beauty)
Decision
Where to open the next 20 stores out of all unclaimed SGP territory
Inputs
Bundle + brand's own list of top-performing store locations (just hex_ids or lat/lng)
Workflow:
1. Convert brand's top-10 store lat/lng → hex_ids via h3.latlng_to_cell(..., 9)
2. Retrieve GCN-64 vectors for those 10 hexes, average → brand_centroid
3. Compute cosine similarity between centroid and all 7,318 hexes
4. Filter out hexes already containing the brand (or tier-1 competitors)
5. Filter for feasibility: actual_cafe_count > 0 OR pred_cafe > threshold
6. Return top 20 ranked by similarity × feasibility score
Output
CSV / GeoJSON of 20 hex_ids + lat/lng + similarity score + neighborhood profile, ready for field-scout validation
Success metric
6-month revenue of new stores opened from the list vs. brand's historical expansion baseline
Risks: (1) Brand's top stores may share PA (planning area) — model will cluster there. Spread centroid across geographic regions. (2) Similarity ignores operational constraints (rent, lease availability, footfall-hour match). Treat as shortlist, not final answer.
2Gap-Driven Site SelectionXGBoost preds●
Retail / F&Bpredictionplanning
Who
Strategy teams at chains (7-Eleven, FairPrice, Toast Box) scouting for under-served areas
Decision
"Where does the model say X should exist, but it doesn't yet?" — finds gaps competitors haven't filled
Inputs
Bundle only
Workflow:
1. For each hex: gap = pred_cafe_coffee – actual_cafe_count
2. Standardize gap (z-score) so we get hexes with structurally-unexpected absence
3. Filter gap_z > 2 (top ~2.5%)
4. Join with population + walkability for secondary ranking
5. Visualize on map: size = gap, color = population density
Output
Map layer of high-gap hexes per category — interactive dashboard or GeoJSON feed
Success metric
(a) Correlation of high-gap hexes with places opened in the 12 months after publication   (b) hit rate if client acts on recommendations
Risks: XGBoost predicts based on physical context; if the model has systematic blind spots (e.g. HDB-specific rules), gaps will be artificial. Validate against holdout set first.
3Amenity Gap Analysis (Policy)XGBoost●●
URA / MND / HDBrolluppolicy
Who
Planning officers deciding where to provision childcare centres, CHAS clinics, hawker stalls, community spaces
Decision
Which of 332 subzones are structurally under-served in specific categories and by how much
Inputs
Bundle + population table (already in data/hex_v10/hex_population.parquet)
Workflow:
1. Per hex: compute deficit = pred_cat – actual_cat for all 24 categories
2. Roll up: subzone_deficit = sum(deficit) / sum(population) per 10K residents
3. Rank 332 subzones by deficit in each category
4. Cross-tab: which subzones have concurrent deficits in ≥ 3 categories (compound priority)
5. Pre-compute & serve as subzone_gap_table.parquet (~30 KB)
Output
332 × 24 gap table + rankings + compound-priority list. Drop-in for URA/HDB dashboards or briefings.
Success metric
Adoption in existing HDB masterplan briefings; agreement with field surveys from residents' associations
Risks: Predictions are based on snapshot; policy decisions have 5–10 year horizons. Pair with SingStat population projections for forward-looking version.
4Comparable Market AnalysisGCN-64●
Real Estatesimilarityappraisal
Who
Real estate appraisers, JLL / CBRE / Savills research teams, shophouse investors
Decision
For a specific property at address X, find 10 hexes with comparable urban character (not just "nearby")
Inputs
Bundle + property address/coords
Workflow:
1. Address → lat/lng → hex_id
2. find_similar(hex_id, k=10) using GCN-64 cosine
3. For each comparable: return parent subzone, PA, and any actual transaction data available
4. Export as comp sheet with similarity score & distance from target
Output
Comparable sheet: 10 hex_ids + similarity scores + metadata, ready for valuation models
Success metric
Appraiser-reported usefulness; blind comparison to human-selected comps
Risks: GCN-64 captures urban character, not property-specific features (building age, condition, view). Use alongside traditional comps, not as a replacement.
5Urban Archetype ClusteringGCN-64●●
Planners / Researchunsupervisedmap layer
Who
URA researchers, academics, anyone needing a neighborhood-type taxonomy for SGP
Decision
Define 10–20 SGP neighborhood archetypes (e.g. "CBD core," "mature HDB," "industrial edge," "tourism cluster") as a segmentation layer for other analyses
Inputs
Bundle only
Workflow:
1. KMeans(n_clusters=15, random_state=42).fit(GCN-64)
2. Characterize each cluster by its centroid's top features (predicted category profile, density, transit access)
3. Name each archetype via LLM or domain expert — publish the vocabulary
4. Attach cluster label to every hex → archetype_labels.parquet
5. Serve as map layer + lookup table
Output
7,318 hexes × 1 archetype_id + named archetype vocabulary (JSON). Re-usable segmentation for any downstream task.
Success metric
(a) Silhouette score / Davies-Bouldin on clustering quality   (b) human-rated coherence of named archetypes
Risks: k=15 is a design choice. Try k=10, 15, 20 and pick what aligns with URA's planning-area vocabulary. Document that archetypes are data-driven, not policy-driven.
6Look-Alike Targeting (Marketing)GCN-64●
Marketing / Adtechsimilaritygeo-fencing
Who
Programmatic ad buyers, OOH (out-of-home) media planners, CRM geo-marketing teams
Decision
Given a set of high-performing campaign zones (known from conversion data), find all SGP hexes with matching character for budget expansion
Inputs
Bundle + campaign conversion data (hex-level or point-level)
Workflow:
1. Map converting customer locations (or campaign-performing zones) → hex_ids
2. Compute conversion-weighted centroid in GCN-64 space
3. Rank all 7,318 hexes by cosine similarity
4. Cap at top-N hexes (e.g. 500) matching target audience size
5. Export as GeoJSON geofences for ad platforms (Meta, Google, DOOH)
Output
Ranked hex list + geofence GeoJSON, audience-size estimate
Success metric
Campaign CAC / CPA in look-alike zones vs. baseline zones. A/B test split.
Risks: Cold-start: if input zones are geographically clustered, look-alikes will be too. Weight by performance not raw count, and diversify seed zones.
715-Minute City Scorecardraw counts + walkability●●
Sustainability / URAcomposite metricdashboard
Who
URA sustainability team, smart-city researchers, town council strategists
Decision
Per hex, how walkably accessible are the 24 daily-life categories? Flagships the "amenity access" metric underlying 15-min-city policy.
Inputs
Bundle + walkability index (already in features)
Workflow:
1. For each hex: count of the 24 categories present in the hex OR its k=1 H3 ring
2. Weight by walkability_index (already in the 391 features)
3. Normalize: score = weighted_cat_coverage / 24 × 100 ∈ [0, 100]
4. Per-category breakdown: which categories are missing in low-scoring hexes?
5. Map layer with choropleth
Output
Per-hex 15-min-city score (0–100) + category-level breakdown + missing-category list
Success metric
Correlation with URA's own accessibility metrics; resident-survey satisfaction scores by subzone
Risks: 400m hex size ≠ 15-min walk. Calibrate ring radius (k=1 ring + walkability) to actual pedestrian speed (~5 km/h → 1.25 km in 15 min).
8Food / Amenity Desert Mappingpred vs actual●●
NGO / Public Healthequitypolicy
Who
Social service agencies (NGOs), public health researchers, HDB estate welfare teams
Decision
Identify hexes where vulnerable populations are structurally cut off from hawker, convenience, clinic, or education access
Inputs
Bundle + population (by age ideally) + MRT/bus proximity (in features)
Workflow:
1. For categories of interest (hawker, convenience, health_medical, education):
  deficit = max(0, pred_cat – actual_cat) × population
2. Weight by walkability (low walkability = deeper desert effect)
3. Composite desert score per hex
4. Highlight hexes with low transit access + high deficit (compound vulnerability)
5. Publish as map + ranked list for council-level interventions
Output
Desert index per hex, top-100 priority list, compound-vulnerability flags
Success metric
Alignment with existing food security research; pilot intervention outcomes (e.g. new hawker stall in top-flagged hex)
Risks: Without income data, can't truly measure vulnerability. For serious equity work, request age + income distribution from SingStat.
9Brand Whitespace AnalysisGCN-64 + filter●●
Brand Strategycompetitiveexpansion
Who
Brand CMOs, expansion teams, competitive intelligence analysts
Decision
Where does Starbucks fit, but isn't yet? Find hexes similar to Starbucks-heavy areas that have zero Starbucks presence (or presence below expected).
Inputs
Bundle + brand presence data (joined from sgp_places_v2.jsonl brand field)
Workflow:
1. Identify hexes containing brand (or tier-1 competitor)
2. Compute brand centroid in GCN-64
3. Rank all hexes by similarity to centroid
4. Filter: hexes with similarity in top 25% AND zero brand presence
5. Secondary rank by size of competitor presence (high = contested, low = whitespace)
6. Export "uncontested whitespace" list
Output
Two lists: (a) whitespace (similar + brand absent + low competition), (b) contested (similar + competitor-heavy)
Success metric
Strategic alignment with brand's stated expansion hypothesis; test via partial rollout
Risks: "Absence" of a brand at 400m granularity is noisy. Use k=1 H3 ring presence, not just the focal hex.

11Delivery / Service Zone DesignGCN-64 + geo●●●
Q-commerce / Logisticsclustering + geo
Who
Ops teams at Grab, foodpanda, Deliveroo, cloud-kitchen companies, local chains with in-house delivery
Decision
Group hexes into 30–50 service zones where each zone has (a) geographic contiguity, (b) similar demand mix, (c) balanced order volume
Inputs
Bundle + geographic centroids + (optionally) historical order volume
Workflow:
1. Use constrained clustering on [GCN-64 features + lat/lng]
2. Add geographic contiguity constraint (each cluster is a connected region)
3. Balance constraint: sum(predicted demand) per cluster within ±20%
4. If historical order data available: calibrate demand weights
5. Output: zone_id per hex + zone summary (area, demand index, dominant archetypes)
Output
Zone boundaries (GeoJSON), zone-summary dashboard, demand-balance report
Success metric
Delivery time variance reduction; utilization rate per zone; driver income parity
Risks: Constrained clustering is computationally expensive and has no closed-form optimum. Use heuristics (spatial k-means + post-hoc boundary repair) or SKATER regionalization.
12New Concept Fit ScoringGCN-64 + mg_* features●●
F&B / Retailscoring
Who
New concept operators, concept-led brands (specialty desserts, boutique fitness, experience retail)
Decision
Where does a new concept fit? Example: a specialty matcha dessert bar wants "high demand_diversity + low comp_pressure + cafe-rich + walkability > 60"
Inputs
Bundle (+ concept profile defined by operator)
Workflow:
1. Operator defines concept profile as a weighted feature vector:
  {demand_diversity: +0.3, mg_cafe_comp_pressure: −0.2, walkability: +0.2, pred_cafe: +0.15, pred_hospitality: +0.15}
2. Normalize weights, compute per-hex score = weighted sum
3. Optionally: find anchor hex (e.g. Tiong Bahru), add similarity to anchor as a factor
4. Rank all 7,318 hexes, return top-20 with full feature-value breakdown
5. Operator validates by visiting top-5 in person
Output
Ranked candidate hexes + feature-value breakdown explaining each score — explainable, not black-box
Success metric
Operator's gut-check acceptance rate; first-year revenue of sites chosen from list
Risks: Operators often lack vocabulary to define "fit." Offer a UI where they pick 3–5 reference hexes; derive the profile automatically.
13Tourism Cluster Detectionmulti-category●●
STB / Tourismproduct dev
Who
Singapore Tourism Board, hospitality chains, destination marketers, tourism-experience startups
Decision
Identify emerging tourism microdistricts. Candidate hexes have joint strength in culture, hospitality, dining, and transit — suggesting a potential tourist cluster not yet on official maps.
Inputs
Bundle only
Workflow:
1. Composite score = w1 × pred_culture_entertainment + w2 × pred_hospitality + w3 × pred_restaurant + w4 × transit_access
2. Rank hexes, take top 500
3. Cluster geographically (DBSCAN on lat/lng) → microdistricts

2. Tier 2 — Works With Bundle + One Data Join 1-day integration

Eight use cases that unlock when bundle joins to one external dataset. Each shows the join + workflow.

14HDB Resale Price Predictionbundle + HDB data●●
Real Estateregression
Who
PropTech platforms, HDB research teams, investment advisors
Decision
Predict resale price per sqft at hex granularity; identify under-priced micro-markets
External data
data/hex_v10/hex_hdb_prices.parquet (already in repo) — HDB resale transactions aggregated to hex
Workflow:
1. Load bundle + hex_hdb_prices, join on hex_id
2. Features = GCN-64 (64 dims) ⊕ pred_residential + pred_transport + pred_education
3. Target = log(price_per_sqft)
4. Train XGBoost with 5-fold CV
5. Predict for all 7,318 hexes → compare to actual
6. Gap (actual – predicted) flags under/over-priced micro-markets
Output
Hex-level price prediction + residual analysis + heat map of valuation gaps
Success metric
CV R² (expected 0.60–0.75); correlation with HDB's own valuation models
Risks: HDB prices have policy-driven boundaries (PMT eligibility, grants) that features don't capture. Residuals may reflect policy, not mispricing.
15Commercial Rent Modelingbundle + rent data●●●
Real Estateregression
Who
Commercial landlords, retail brands, leasing agents
Decision
Predict prime shop rent psf by hex; detect rent gaps vs. commercial potential
External data
JLL / CBRE / Savills rent benchmarks (not in repo; would need to scrape or license)
Workflow:
1. Geocode rent observations to hex_ids
2. Ridge or XGBoost regression: y = rent_psf, X = GCN-64 + 24 preds + walkability + transit_access
3. Predict for all commercial-zoned hexes
4. Rent-gap = actual_rent – predicted_rent
5. Negative gap → under-priced (retail opportunity); positive gap → over-priced
6. Deliver as lease-negotiation tool
Output
Per-hex rent prediction + gap + confidence interval
Success metric
CV R²; agreement with licensed-agent reports
Risks: Commercial rent data is sparse and heterogeneous (office vs retail vs F&B). Split training by use type. Report CIs, not point predictions.
16Footfall / Dwell-Time Predictionbundle + telco / GPS●●●
Retail Opstransfer learning
Who
Mall operators, retail chains with sensors, ad network resellers
Decision
Predict pedestrian footfall and dwell-time per hex per time-of-day, transfer from instrumented hexes to all 7,318
External data
Telco-aggregated visits (M1, Singtel), or licensed Grab/Uber mobility traces, or mall foot-traffic sensors
Workflow:
1. Instrumented hexes: observed footfall_by_hour
2. Train MLP / XGBoost: y = log(footfall), X = GCN-64 + pred_* + transit + population
3. Predict for the 6,500+ uninstrumented hexes
4. Decompose by hour-of-day using shared temporal basis functions
5. Serve as hourly footfall heatmap
Output
Per-hex footfall by hour; uncertainty bands from CV
Success metric
Hold-out R²; temporal correlation with spot-sensor validation
Risks: Transfer learning assumes instrumented-vs-uninstrumented hexes are drawn from the same distribution. If instrumentation is only in malls, prediction on HDB hexes is extrapolation.

Who
Urban sociologists, heritage-protection NGOs, REIT analysts, HDB equity researchers
Decision

External data
Requires ≥ 2 bundle snapshots (current = t=0; at t=1, rebuild from fresh place data)

4. Integration Patterns (How Consumers Actually Use This)

Every use case falls into one of four integration patterns. Consumers can mix and match.

Pattern A — Offline Analytics (batch)

Consumer downloads the parquet, joins to their own data, runs analysis in pandas/R/SQL. Most flexible, zero latency constraint.

pull parquet (2.8 MB)→join in pandas→ranking / regression / cluster→export CSV or map

Fits: #2, #3, #5, #8, #10, #11, #13, #14, #15, #17–#21 (14 use cases)

import pandas as pd
bundle = pd.read_parquet("hex_shareable_bundle.parquet")
# ... your analysis in pure pandas

Pattern B — Interactive Lookup (real-time)

Consumer calls an HTTP endpoint for specific queries (map click, kNN search, bounding-box). Best for web and mobile UIs.

user clicks map tile→GET /hex/{id}→server returns JSON→render tooltip/panel

Fits: #1, #4, #6, #9, #12, #13 (6 use cases)

fetch("/api/hex/896520db3a/similar?k=10")
  .then(r => r.json())
  .then(results => renderMap(results))

Pattern C — Training Features (ML pipeline)

Consumer uses GCN-64 or raw 391 features as inputs to their own ML models (gradient boosting, neural nets, survival analysis).

load features→join target (price, claims, footfall)→train XGBoost/MLP→predict / deploy

Fits: #14, #15, #16, #17, #18, #19, #20 (7 use cases)

import xgboost as xgb
emb = bundle[[f"g{i}" for i in range(64)]].values
y = my_data["target"].values
model = xgb.XGBRegressor().fit(emb, y)

Pattern D — Pre-Computed Rollups (dashboards)

For common queries (subzone rollups, archetype labels), ship a pre-computed artifact that answers the question in one read. No compute by consumer.

consumer loads subzone_gap_table.parquet→already ranked→publish dashboard

Fits: #3, #5, #7, #8, #10 (5 use cases)

5. Worked Example — Site Selection End-to-End (Use Case #1)

Concrete walkthrough: a boutique specialty coffee brand with 8 stores wants to open 12 more. Here's the full pipeline using the bundle.

Step 1 — Define the anchor set

# Brand provides 8 top-performing store lat/lng pairs
import h3, pandas as pd, numpy as np

anchors = pd.DataFrame([
    {"name": "Tiong Bahru", "lat": 1.2846, "lng": 103.8306},
    {"name": "Duxton",      "lat": 1.2792, "lng": 103.8421},
    # ... 6 more top stores
])
anchors["hex_id"] = anchors.apply(lambda r: h3.latlng_to_cell(r.lat, r.lng, 9), axis=1)

Step 2 — Compute brand centroid in GCN-64

bundle = pd.read_parquet("hex_shareable_bundle.parquet")
emb_cols = [f"g{i}" for i in range(64)]
anchor_embs = bundle.set_index("hex_id").loc[anchors.hex_id, emb_cols].values
centroid = anchor_embs.mean(axis=0)

Step 3 — Rank all hexes by similarity + feasibility

Z = bundle[emb_cols].values
sims = (Z @ centroid) / (np.linalg.norm(Z, axis=1) * np.linalg.norm(centroid) + 1e-9)
bundle["similarity"] = sims

# Feasibility: hex must have commerce already present and not already hold the brand
bundle["feasible"] = (
    (bundle["pred_cafe_coffee"] > 1.5) &
    (~bundle["hex_id"].isin(anchors.hex_id))
)
bundle["score"] = bundle.similarity * bundle.feasible.astype(float)

Step 4 — Return top 12 with explanations

top12 = bundle.sort_values("score", ascending=False).head(12)
top12[["hex_id", "lat", "lng", "parent_subzone",
      "pred_cafe_coffee", "similarity"]]

Step 5 — Deliver as GeoJSON for field scouting

top12_geojson = {
    "type": "FeatureCollection",
    "features": [{
        "type": "Feature",
        "geometry": {"type": "Point", "coordinates": [r.lng, r.lat]},
        "properties": {"hex_id": r.hex_id, "rank": i+1, "sim": float(r.similarity)}
    } for i, r in top12.iterrows()]
}

Track A — Bulk Download must-have

Covers 13 of 21 Tier-1/2 use cases. Zero infrastructure cost.

  • • Direct-URL mirror on rwm-server nginx
  • • pip install sgp-hex-atlas with parquet + helpers

Users: researchers, data scientists, modelers, anyone doing pandas joins.

Track B — Lightweight API nice-to-have

4 endpoints, FastAPI on rwm-server. Bundle pre-loaded in RAM. < 5 ms p50.

  • • GET /hex/{id} → full record
  • • GET /hex/{id}/similar?k=10 → kNN
  • • GET /hex/{id}/predictions → 24 counts
  • • POST /bbox → hexes in a viewport

Users: web UIs, dashboards, non-Python integrations.
